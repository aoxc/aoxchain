# AOXCVM Test Strategy and Phase-3 Closure Plan

This document defines the test posture required to move AOXCVM from a **working phase-3 entry implementation** to a **production-grade phase-3 runtime**.

## 1) Verified Baseline (as of 2026-04-02)

`cargo test -p aoxcvm` currently demonstrates a broad, executable core surface:

- 135 unit tests passed.
- 5 system compatibility tests passed.
- 4 phase-2 runtime integration tests passed.
- 0 failures.

The current suite already validates these operational areas:

- Executor behavior (deterministic arithmetic, OOG canonical trap, malformed immediate rejection).
- Opcode policy (fail-closed unknown/reserved opcodes and stable opcode roundtrip).
- Memory/stack/PC mechanics (bounds checks, zero-fill expansion, explicit transitions).
- Host journal semantics (atomic commit/reject paths, rollback, checkpoint behavior, lane conflict handling).
- Bytecode and determinism verifier foundations.
- Receipt/commitment/proof/state-root stability checks.
- Admission and runtime lane/policy enforcement.
- Constitutional runtime and audit reason-code mapping paths.

## 2) What Is Still Missing for Production-Grade Phase-3

AOXCVM has crossed from architecture-only status into executable reality. Remaining gaps are not “core missing,” but **evidence depth and coverage breadth**.

### P0 (Phase-3 completion blockers)

1. **Adversarial validation depth**
   - Add fuzz targets (`cargo fuzz`) for bytecode parser, verifier, syscall argument decoding, receipt decoding, and PQ witness handling.
   - Add property-based tests (`proptest`/`arbitrary`) for determinism invariants, gas monotonicity, and rollback invariants.

2. **Cross-platform determinism matrix**
   - Validate identical outputs across architecture/OS/toolchain combinations.
   - Enforce release-vs-debug parity checks for deterministic outputs and failure receipts.

3. **Verifier breadth expansion**
   - Add bytecode lifecycle rejection matrix: section ordering, duplicate sections, size limits, invalid branch targets, malformed pools, and canonical serialization vectors.

4. **Call semantics hardening**
   - Add nested call tests: depth limits, revert bubbling, gas forwarding semantics, return-data propagation, and frame-isolation invariants.

5. **Gas-economics evidence**
   - Add opcode/memory/syscall benchmark suites and DoS envelope analysis with p50/p95/p99 targets.

### P1 (Production readiness blockers)

1. **System-level integration expansion**
   - Multi-tx/block execution ordering determinism.
   - Admission → execution → receipt pipeline end-to-end coverage.

2. **Syscall ABI governance**
   - ABI versioned fixtures, payload bounds, and canonical encoding golden tests.

3. **PQ execution integration depth**
   - Mixed-profile runtime stress tests and malformed witness stress paths tied to execution economics.

4. **Release-gating CI evidence**
   - Mandatory fuzz stages, determinism matrix gates, benchmark regression gates, and retained evidence artifacts.

5. **Audit bundle readiness**
   - Threat model, invariant map, gas justification memo, failure taxonomy, known limitations register, and versioned fixture pack.

### P2 (Differentiation and long-term governance)

- Deterministic execution fingerprinting for validator heterogeneity diagnostics.
- Formal compatibility/deprecation framework and migration rehearsal suite.
- Governance-backed gas schedule recalibration workflow with benchmark evidence.
- Constitutional provenance standardization as canonical receipt augmentation.

## 3) Required Evidence Artifacts

Phase-3 closure must be demonstrated with reproducible artifacts, not only passing local tests.

Required artifacts:

- Determinism matrix report (platforms/toolchains/results digest).
- Fuzz corpus retention with minimized crash/repro fixtures.
- Benchmark dossier (opcode/syscall/memory/PQ verification costs + hardware classes).
- Compatibility/serialization vectors and rejection matrix outputs.
- Audit evidence bundle generated by a stable release command.

## 4) Exit Criteria for “Production-Grade Phase-3”

AOXCVM should be labeled production-grade phase-3 only when all of the following are true:

1. P0, P1 suites are implemented and release-gated in CI.
2. Determinism parity is proven across the supported validator matrix.
3. Gas schedule is benchmark-calibrated and DoS-envelope justified.
4. Call/storage/syscall semantics have broad adversarial and integration coverage.
5. A versioned audit evidence bundle is generated for each release candidate.

## 5) Current Status Statement

**Current assessment:** AOXCVM is no longer a scaffold-only VM. It is a working VM core with runtime governance and verifier/admission surfaces, currently at **late phase-2 / phase-3 entry with partial constitutional phase-3 completion**.

**Practical conclusion:** Production-grade phase-3 closure now depends primarily on adversarial testing depth, cross-platform determinism proof, economic calibration, and release evidence discipline.
