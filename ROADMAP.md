# AOXChain Roadmap (Execution Baseline)

This roadmap defines the canonical, evidence-gated delivery plan.

## Planning Model

The roadmap is milestone-driven, not calendar-driven:

- phase closure requires objective evidence,
- checklist completion requires attributable artifacts,
- promotion decisions are governance outcomes, not timeline assumptions.

## Program Goal

Deliver a deterministic Layer-1 architecture that is:

- classically secure in current operation,
- post-quantum-primary by governed transition,
- migration-safe by protocol design,
- evidence-gated at each promotion boundary.

No item is complete without reproducible command and artifact evidence.

## Non-Negotiable Constraints

1. No claim of absolute or permanent cryptographic security.
2. No hidden fallback bypassing profile-governed crypto selection.
3. No architecture-sensitive merge without synchronized specs/tests/docs.
4. No readiness declaration without retained evidence artifacts.

## Phase 0 — Governance Alignment

Objective: establish one authoritative planning and terminology surface.

Exit criteria:

- root governance docs are synchronized,
- terminology is stable across documents,
- no root-level policy conflicts remain.

## Phase 1 — Authority and PQ Specification

Objective: finalize canonical authority model before implementation expansion.

Exit criteria:

- actor taxonomy and proof schema fixed,
- replay semantics and profile lifecycle documented,
- wallet/node key-domain lifecycle contract specified.

## Phase 2 — State and Serialization Contract

Objective: lock deterministic state and wire formats.

Exit criteria:

- versioned schema and serialization vectors are fixed,
- migration transaction types defined,
- rollback posture documented.

## Phase 3 — Validation Kernel

Objective: implement deterministic, profile-aware, fail-closed validation.

Exit criteria:

- verify-first boundary enforced,
- downgrade/mismatch rejection deterministic,
- adversarial malformed/replay corpus in place.

## Phase 4 — Consensus and Governance Integration

Objective: integrate authority kernel into consensus and governance execution paths.

Exit criteria:

- validator/governance auth fully policy-driven,
- emergency rotation drills complete,
- profile negotiation hardening and telemetry validated.

## Phase 5 — Testnet Hardening

Objective: operate testnet under production-style controls.

Exit criteria:

- required gates pass across defined window,
- rollback and migration drills are artifacted,
- unresolved critical findings have owner and timeline.

## Phase 6 — Mainnet Activation Decision

Objective: permit activation only after governed acceptance of residual risk.

Exit criteria:

- activation dossier complete,
- rollback constraints tested,
- engineering and operations sign-off recorded,
- known limitations and mitigations declared.

## Program-Wide Always-On Criteria

- deterministic behavior for identical canonical inputs,
- independent and auditable policy/recovery roots,
- domain-separated replay protection,
- explicit profile lifecycle governance,
- synchronized implementation and documentation.
