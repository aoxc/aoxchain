# ROADMAP.md

Repository-level execution roadmap for AOXChain.

## Current Gap Snapshot (April 1, 2026)

The repository has strong governance and operator surfaces, but release posture is still blocked by several implementation gaps:

- runtime admission does not yet enforce registry validation before settlement;
- proof/finality verifier interfaces are not yet fully implemented across declared finality classes;
- replay-ledger persistence and conflict forensics are not yet shipped as an operator-verifiable flow;
- interoperability conformance and adversarial CI gates are not yet release-blocking.

Until these items are complete and evidence-backed, readiness should be treated as **development-grade** rather than production-grade.

## Phase A — Deterministic kernel baseline
- [x] Establish AOXCLang language-family model.
- [x] Add adapter contract and relay-envelope validation surface.
- [x] Add registry-level adapter conformance entrypoint.
- [ ] Wire registry validation into runtime admission path before settlement.

**Definition of done (remaining item):**
- runtime admission path rejects non-conformant registry entries with fail-closed behavior;
- settlement path cannot be reached when registry validation fails;
- targeted regression tests cover admission reject/accept boundaries;
- release evidence includes command output proving the gate is active.

## Phase B — Interoperability safety hardening
- [ ] Define proof verification interfaces per finality class.
- [ ] Add replay-ledger persistence model and conflict forensics output.
- [ ] Introduce canonical relay intent envelope versioning policy.
- [ ] Add deterministic cross-adapter conformance matrix tests.

**Definition of done (phase):**
- each supported finality class has an explicit verifier interface contract and test coverage;
- replay-ledger persistence is deterministic, queryable, and includes conflict forensics artifacts;
- relay intent envelope versioning is documented with compatibility and migration rules;
- conformance matrix tests are deterministic and run in default CI quality workflows.

## Phase C — Adversarial validation and quality gates
- [ ] Add CI adversarial scenarios (replay, malformed proof, delayed finality).
- [ ] Add reorg simulation matrix across representative language families.
- [ ] Add fuzz targets for relay-envelope parser/validator boundaries.
- [ ] Block releases on failed conformance and replay-safety checks.

**Definition of done (phase):**
- adversarial scenarios execute in CI and are retained in evidence artifacts;
- reorg simulation matrix has deterministic fixtures and documented pass/fail criteria;
- parser/validator fuzz targets are reproducible and integrated into quality gates;
- release automation fails closed on conformance or replay-safety regressions.

## Phase D — Operator and production posture
- [ ] Publish readiness scorecards (kernel, adapters, proofs, replay safety).
- [ ] Add telemetry for adapter proof latency and reject causes.
- [ ] Add release evidence artifact bundle for interoperability checks.
- [ ] Finalize migration/versioning playbook for policy surface changes.

**Definition of done (phase):**
- scorecards are generated from repository-governed inputs and stored in `artifacts/`;
- telemetry schema is documented and includes reject-reason taxonomy;
- interoperability evidence bundle is generated by a stable command path and audited in review;
- migration/versioning playbook defines compatibility classes, operator actions, and rollback controls.

## Ongoing discipline
- [ ] Keep documentation aligned with implemented behavior in the same PR.
- [ ] Preserve deterministic behavior and compatibility-sensitive identifiers.
- [ ] Require explicit migration notes when changing protocol-facing semantics.

## Release Blockers (must close before production-readiness claims)

1. Runtime admission enforcement for registry validation.
2. Finality-class verifier interfaces with deterministic tests.
3. Replay-ledger persistence plus operator forensics surface.
4. CI-enforced adversarial and conformance gates.
5. Interoperability evidence bundle integrated into release flow.
