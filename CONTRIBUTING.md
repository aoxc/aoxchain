# Contributing

## Rules
- Keep changes minimal, explicit, and reviewable.
- Prefer deterministic behavior and auditable logic over convenience shortcuts.
- Document architectural intent for non-trivial changes.

## Do Not
- Do not merge dead code, placeholders, silent fallbacks, or TODO-only behavior.
- Do not introduce implicit breaking changes to state, protocol, or API contracts.

## Testing
- Run formatting, linting, and relevant unit/integration tests before opening a PR.
- Add regression tests for bug fixes and behavior changes.

## Review
Reviewers expect clear scope, risk notes, and explicit validation evidence for production-leaning changes.
